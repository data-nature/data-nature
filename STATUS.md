# Project Status — data-nature

## What this project is

A data science tool that monitors vegetation health and heat stress across 8 sites in Northern Israel.
It pulls real satellite data from NASA (via Google Earth Engine), runs statistical analysis on it,
and shows the results on a web dashboard built with Streamlit.

The 8 monitored sites cover different land types: forests, farmland, wetlands, urban areas, and arid zones.

---

## What works right now

The project has a running Streamlit dashboard with 7 pages (heatmap, anomalies, simulator, forecasting,
optimization, reports, and an AI Q&A page). It was previously running on mock/synthetic data.
The backend Python package is now being built out with real logic.

**Purpose:** Connect to Google Earth Engine and pull real monthly temperature and vegetation data
for all 8 sites, going back to the year 2000.

**Files added:**

- `src/data_nature/ingest/earth_engine.py`
  The main ingest module. Contains three things:

### DN-A2 — Statistical analysis engine

**Purpose:** Given the satellite data, detect temperature anomalies per site and run statistical
tests to understand the relationship between vegetation and temperature.

**Files added:**

- `src/data_nature/stats/anomaly.py`
  Two functions:
  - `compute_zscores(df)` — for each site and each calendar month, computes the long-run historical
    average temperature (the "baseline") and measures how far the current reading is from that average
    in standard deviations. A z-score of 0 means normal, a high positive z-score means unusually hot.
  - `detect_anomalies(df, thresholds)` — filters the data to only the anomalous readings and labels
    each one: **Warning** (z ≥ 1.5), **Severe** (z ≥ 2.5), or **Critical** (z ≥ 3.5).
    Returns a table in the exact format the dashboard expects (`anomalies.csv` schema).

- `src/data_nature/stats/regression.py`
  Two functions:
  - `fit_ndvi_lst(df)` — fits a linear regression to answer: "does more vegetation predict lower
    temperature?" Runs both a simple model (temperature vs vegetation only) and a richer model
    that also accounts for land cover type and month. Returns the slope, R², and p-values.
    The expected finding is a **negative coefficient** — greener sites run cooler.
  - `compare_site_types(df)` — runs a one-way ANOVA test to answer the research question:
    "is mean temperature statistically different across land cover types (forest vs arid vs urban etc.)?"
    Also runs pairwise t-tests between every pair of land types.
    H0 = all land types have the same mean temperature. H1 = at least one differs.

- `src/data_nature/stats/__init__.py`
  Makes all four functions importable from `data_nature.stats`.

- `tests/test_stats.py`
  34 unit tests. Key scenarios covered:
  - A synthetic 20°C temperature spike is correctly flagged as Critical.
  - A perfectly stable series produces zero anomalies.
  - The NDVI regression coefficient comes out negative (scientifically expected).
  - ANOVA correctly rejects H0 when groups have different means, and does not reject it when they are the same.

## What Alaa did in this sprint

### DN-B1 — EDA, data-quality audit & baseline computation (merged, PR #1)

**Purpose:** Verify the real dataset Ahmad prepared, validate it matches the frozen data contract,
and compute the per-site per-month historical baselines that the z-score engine consumes.

**Files added:**

- `src/data_nature/stats/baselines.py`
  Two main functions:
  - `validate_schema(data_dir)` — checks every processed CSV against the frozen data contract
    (column names and order). Raises an error if a present file has missing columns, warns and
    skips files not yet generated by other tickets. Currently validates 4 of 7 contract files
    (anomalies, forecast, and metrics are built in later tickets).
  - `compute_baselines(df, min_years=3)` — expanding leave-one-out window per (site, month):
    each year's baseline is computed from all _prior_ years only, so the current year never
    contaminates its own baseline. This is critical for z-score validity — you can't use
    today's reading to define what "normal" means for today.
  - `build_and_save_baselines()` — one-call helper that loads site_monthly, runs the above,
    and saves the result to `data/processed/site_baselines.csv`.
  - CLI: `python -m data_nature.stats.baselines` runs schema validation then generates the CSV.

- `data/processed/site_baselines.csv`
  Pre-computed baseline table — 2,528 rows covering all 8 sites × 12 months × 26 years.
  Ready to consume directly without re-running anything.

- `notebooks/eda.ipynb`
  Exploratory data analysis notebook covering distributions per site and land-cover type,
  NDVI–LST scatter, missingness map, seasonality patterns, and outlier checks for all 8 sites.

- `tests/test_baselines.py`
  20 tests across 5 classes: exact numeric correctness on known synthetic series, edge cases
  (early NaN rows, single-prior-year std, missing required column), schema validation (happy
  path, missing file, missing column, extra columns), multi-site and multi-month independence,
  and real-data integration (shape, all 8 sites present, no future data leakage).

### DN-B2 — ML forecast models: Random Forest + Gradient Boosting (PR #2)

**Purpose:** Train two machine learning models to forecast LST 7 months ahead per site,
produce uncertainty bands, and write the metrics and forecast tables the Forecast page will consume.

**Files added:**

- `src/data_nature/models/forecast.py`
  The core forecasting module. Contains:
  - `_build_features(df)` — engineers 6 predictive features from the raw monthly data:
    previous month's LST (`lst_lag1`), same month last year (`lst_lag12`), 3-month rolling
    mean and std (`lst_roll3_mean`, `lst_roll3_std`), NDVI, and an integer site encoding.
    All lag operations stay within each site — no cross-site data leakage.
  - `train_forecasters(df, train_end_year=2023)` — trains Random Forest (200 trees, max depth 8)
    and Gradient Boosting (200 estimators, learning rate 0.05) on 2000–2023 data, holding out
    2024–2026 as a clean test split for honest metric computation.
  - `forecast(models, site, horizon=7)` — recursive multi-step forecast: predicts one month
    ahead, feeds that prediction as the next lag input, repeats for `horizon` steps. Returns
    a DataFrame matching the `lst_forecast.csv` schema exactly (date, site, model,
    lst_forecast, lst_low, lst_high). Uncertainty bands use tree prediction variance for
    Random Forest and a rolling-std fraction for Gradient Boosting.
  - `compute_metrics(models)` — MAE, RMSE, R² per site and overall, computed strictly on the
    held-out test split. Returns a DataFrame matching the `model_metrics.csv` schema.
  - `run_full_pipeline()` — end-to-end helper: loads data, trains both models, forecasts all
    8 sites, saves `lst_forecast.csv` and `model_metrics.csv` to `data/processed/`.
  - CLI: `python -m data_nature.models.forecast` runs the full pipeline.

- `src/data_nature/models/__init__.py`
  Exports `train_forecasters`, `forecast`, `compute_metrics`, and `run_full_pipeline` so other
  modules can import them with `from data_nature.models import forecast`.

- `tests/test_forecast.py`
  24 tests across 6 classes: feature engineering correctness (lag values, no cross-site leakage),
  training outputs (all models fitted, label encoder covers all sites), forecast schema and row
  count, uncertainty band validity (low ≤ forecast ≤ high always), metrics computed on held-out
  split (not training data), site isolation, and real-data integration (all 8 sites forecastable,
  R² > 0 on real data).
